🔹 STEP 4 — Basic Memoization (Value Caching)

What was added

-In-memory cache ({})
-if (key in cache) check

What it solved

-Repeated requests over time
-Sequential calls reuse cached data

Where you got stuck

You initially:

-used a for...in loop
-returned the wrong cached value

Fix

-Cache lookup must be direct
-Cache is a map, not a list

Rule learned

Memoization checks “THIS key”, not “ANY key”.
----------------------------------------------------------------------------------------
🔹 STEP 5 — Failure Safety

This was your biggest conceptual block

Problem discovered

-Errors were caught but not re-thrown
-Function returned undefined
-Memoization cached undefined
-Failure became permanent

Why this is dangerous

A temporary failure becomes a permanent bug.

Fix

-throw error in loadData
-throw error in memoization
-Never cache on failure

Rule learned

Errors must propagate upward until handled.
Never convert errors into values.

This step unlocked everything else.

🔹 STEP 6 — In-Flight Request Deduplication

This is where real-world thinking begins

Your doubt (valid)

“Is this really needed? JavaScript is single-threaded.”

Clarification

JS is single-threaded

Async operations overlap

Multiple callers can start before cache is filled

Problem
memoizedLoadData(1)
memoizedLoadData(1)
memoizedLoadData(1)


→ multiple fetches start before cache is written

Fix

Cache the Promise immediately

Share the same Promise among callers

Rule learned

Cache the work, not just the result.

This solved concurrency waste.

🔹 STEP 7 — TTL (Cache Expiration)

Final foundational step

Problem

Cached data lived forever

Real data changes

Stale UI bugs

Fix

Add timestamp to cache

Invalidate after TTL

Refetch when expired

Rule learned

Do not trust cached data forever.

This added time awareness to caching.

🧠 Key takeaways (lock these)

-Each function call gets its own Execution Context
-cache lives because of closure
-Async functions pause, not disappear
-Rejected Promise throws back into await
-Cache is deleted on failure → retry possible
-Execution stops unless caller catches error
